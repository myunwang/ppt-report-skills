# Architecture and Build Mechanism

## Project Layout

```
my-report/
├── my-report.html             ← final output (generated by build.py)
├── my-report.pdf              ← generated by export_pdf.py (optional)
├── build.py                   ← merge script
├── export_pdf.py              ← PDF export script
├── xlsx2json.py               ← Excel/CSV → JSON converter (called automatically by build, can also run standalone)
└── src/
    ├── shell.html             ← HTML skeleton, with {{STYLES}} {{SLIDES}} {{SCRIPTS}} placeholders
    ├── styles/
    │   ├── common.css         ← global styles (includes theme variables, component library)
    │   ├── slide-1.css        ← slide 1 specific styles (can be empty)
    │   └── slide-N.css
    ├── slides/
    │   ├── slide-1.html       ← slide 1 .slide div content (without <html>/<body>)
    │   └── slide-N.html
    ├── scripts/
    │   ├── common.js          ← adaptive scaling + navigation + ECharts helper
    │   ├── slide-1.js         ← must export an initSlide1() function
    │   └── slide-N.js
    └── data/                  ← choose freely among three formats, build.py auto-converts to JSON and injects
        ├── slide-1.xlsx       ← recommended: edit data daily with Excel
        ├── slide-2.csv        ← single-table data
        └── slide-3.json       ← complex nested structure / machine-generated
```

## How build.py Works

```python
# 1. Concatenate all styles/*.css → {{STYLES}}
# 2. Concatenate all slides/*.html → {{SLIDES}} (auto-adds <!-- SLIDE N --> comments)
# 3. Concatenate all scripts/*.js → {{SCRIPTS}}
# 4. Write into shell.html placeholders → output final HTML
```

**Why do it this way?**
- Per-slide changes only touch 1~3 small files, each file ≤ 200 lines → editor/Claude context-friendly, token-efficient
- The final output is a single HTML file (no external dependencies, can be shared directly via email/IM)
- When collaborating with multiple people, work can be divided by slide without conflicts

## Data Separation Mechanism

**Core idea**: render code (HTML/JS) and data (xlsx/csv/json) are fully decoupled. Next time you update the data, only change the data file and re-run `python3 build.py` — **not a single line of render code needs to change**.

### Supported Data Formats

| Format | Use Case | Conversion Result |
|---|---|---|
| `.xlsx` | **Recommended for daily use** — data is already in Excel | each sheet → a top-level JSON key, each sheet's content → `[{col: val}, ...]` |
| `.csv` | single-table data | the whole file → `[{col: val}, ...]` |
| `.json` | machine-generated / complex nested structure | read directly |

**Priority**: `.json > .xlsx > .csv` (only one is taken per slide number).

### xlsx / csv Conventions

1. **Row 1 = header**, each subsequent row = data
2. **Each row becomes an object**: `{header column: cell value}`
3. **Automatic type inference**: a purely numeric cell → number, everything else stays string
4. **Filter convention**: sheet names / column names starting with `_` are skipped (used for notes, helper calculation columns)
5. **Empty rows are skipped automatically**

Example: `src/data/slide-3.xlsx` contains two sheets —

| sheet `kpis` | label | value | unit |
|---|---|---|---|
| | DAU | 12.4 | 万 |
| | ARR | 48 | M USD |

| sheet `trend` | week | na | eu |
|---|---|---|---|
| | W1 | 12.1 | 8.4 |
| | W2 | 12.4 | 8.6 |

After conversion, `window.__DATA_3__` is:

```json
{
  "kpis":  [{"label": "DAU", "value": 12.4, "unit": "万"}, {"label": "ARR", "value": 48, "unit": "M USD"}],
  "trend": [{"week": "W1", "na": 12.1, "eu": 8.4}, {"week": "W2", "na": 12.4, "eu": 8.6}]
}
```

### build.py Injection Flow

```
src/data/slide-N.xlsx    ← edited by user
        ↓ build.py calls xlsx2json.xlsx_to_dict(path)
{kpis: [...], trend: [...]}    ← Python dict
        ↓ json.dumps + wrapped as window.__DATA_N__ = ...;
in the final HTML's <script>: window.__DATA_N__ = {"kpis": [...], "trend": [...]};
        ↓ initSlideN() reads via window.__DATA_N__
ECharts / DOM render
```

In `scripts/slide-N.js` you only need:

```js
function initSlideN() {
  const D = window.__DATA_N__;
  // D.kpis = [...], D.trend = [...]
  // just call renderXxx() or initChart()
}
```

### Standalone Debugging

```bash
# Convert a single file (output to stdout)
python3 xlsx2json.py src/data/slide-3.xlsx

# Write JSON (so future builds will prefer the JSON)
python3 xlsx2json.py src/data/slide-3.xlsx > src/data/slide-3.json

# Convert the whole directory
python3 xlsx2json.py src/data/
```

### When to Use Which Format

- **Use xlsx for most scenarios**: business data is already in Excel, just drop it in
- **Use csv for simple one-dimensional data**: csv files exported from BI / databases
- **Use json for nested structures / auto-generated data**: e.g. markLine config (nested objects), complex structures pulled from an API — these are not elegant to express in xlsx, so writing JSON directly is more convenient

### Runtime fetch (optional, not recommended)

Instead of inlining via build.py, the browser does a runtime `fetch('./data/slide-5.json')`, suitable for frequently changing data + server hosting. Note that when opened via `file://`, CORS will block it, so build inlining is the default.

## Adaptive Scaling (Core Constraint)

**All elements are sized against a 1600×900 pixel design canvas**. When the browser window size changes, the entire canvas is uniformly scaled via CSS `transform: scale(--fit)`, so it is always fully visible and never cropped.

```css
:root {
  --design-w: 1600px;
  --design-h: 900px;
  --fit: 1; /* computed and written by JS */
  --nav-reserve: 88px; /* reserved space for the bottom nav bar */
}
.canvas {
  position: absolute; top: 50%; left: 50%;
  width: var(--design-w);
  height: var(--design-h);
  transform: translate(-50%, -50%) scale(var(--fit));
  transform-origin: center center;
}
```

```js
function fitCanvas() {
  const w = window.innerWidth;
  const h = Math.max(0, window.innerHeight - 88);
  const scale = Math.min(w / 1600, h / 900);
  document.documentElement.style.setProperty('--fit', scale.toFixed(4));
}
window.addEventListener('resize', fitCanvas);
window.addEventListener('DOMContentLoaded', fitCanvas);
```

**Side effects of this constraint (must be followed)**:
- ❌ **Do not use** `vw / vh / %` for font size or spacing → it would double-scale with the overall scale
- ✅ Use `px` everywhere, measured against the 1600×900 design canvas
- ✅ Font size also uses `px` (already in the `--fs-*` variables)
- ✅ Chart `font.size` and `borderWidth` also use px design-canvas values

## Navigation and Chart Initialization

`common.js` provides:
- `goTo(idx)` / `go(±1)` — switch slides
- Keyboard ←/→/↑/↓ to page through
- Bottom dots + "1 / 6" counter
- `initCharts(idx)` automatically calls `window.initSlide{N}()`, initializing each slide only once

Each slide's JS must export `function initSlide{N}() { ... }` (even if the slide has no chart, it must have an empty function, otherwise PDF export will not find it).

## PDF Export (export_pdf.py)

- Uses Playwright headless Chromium, viewport = 1600×900, device_scale_factor=2 (@2x, sharp text)
- Injects CSS override: `--fit:1` + hides the nav bar
- Disables Chart.js animation → the chart is fully rendered before the screenshot
- Activates slides one by one → waits for the chart canvas to have pixels → captures PNG
- `img2pdf` merges into a 13.33×7.5 inch (standard 16:9 presentation size) PDF

## Font Loading

`shell.html` imports:
- **Noto Sans SC** — body Chinese text (300/400/500/700/900)
- **JetBrains Mono** — numbers / code / labels
- **Bebas Neue** — large numbers (optional, only used for the "5 items" 64px large numbers on overview-style slides)

If you change fonts when switching themes, just import in that theme's CSS section.
