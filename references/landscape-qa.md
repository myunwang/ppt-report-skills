# Landscape / 结构图类页面 质量自检与优化

> 适用：`landscape-map` 模板那类页面 —— 三层 tier × 多卡片 × 卡内 chip
> （圆角胶囊小块）的纯 CSS 结构图（产业图谱 / 竞品全景 / 客户分层 /
> 系统分层架构）。
>
> 这套清单来自十几轮反复失败的真实教训。生成这类页面后，**逐条对照
> 执行，不要跳过、不要凭感觉判断"应该没问题"**。

## 一、踩坑黑名单（最重要 —— 这些坑反复踩过）

1. **绝不写 JS 测卡片尺寸去反推 chip 字号做"自适应铺满"**。
   cqh / 二分逼近 / 全局系数 / logo 异步重算这条路是死结，试了十几轮全失败。
   chip 字号用 **CSS 固定值**（如 15px / 嵌套 13px），靠数据规范保证不截断。
   JS 只允许做两件事：① logo 注入 ② 统一 chip 宽度（不碰字号）。

2. **绝不只截单个 tier 的局部图就判"通过"**。三层挤一起、整体偏上、下方
   大片留白、虚线错位 —— 这些全局问题在单 tier 局部图里**永远看不出来**。
   **每次必须看完整全页截图**（1600×900 完整 clip），用肉眼确认。

3. **绝不轻信自己写的 scrollWidth / Range 截断检测**。在这套嵌套
   flex/grid + 异步 logo 结构里，自动检测与肉眼**系统性不一致**，
   多次"数字全绿"被全图打脸。检测数字仅供参考，**以全图肉眼为准**。

4. **模板注释头 / 任何说明文字里绝不出现裸 `<style>` 字面量**。
   否则正则提取 style 块时会从注释里那个 `<style>` 字样开始误抓，
   把大段中文说明当 CSS → CSS 解析崩溃 → 规则全失效 → 三层挤一起。
   提取时用 `re.findall(r'<style>(.*?)</style>', s, re.S)[-1]`（取最后一对）。

5. **截图前绝不只用固定 `wait_for_timeout`**。logo 是异步加载的，
   必须**轮询所有 `<img>.complete` 都为 true**（最多等 20s）再截，
   否则截到一堆没加载完的空 logo。

6. **英文长词 / "A / B" 合并项绝不硬塞一个 chip**。从数据层就拆/简写
   （见下方数据规范），渲染层不负责"塞下超长文案"。

## 二、数据规范清单（生成前 / 生成中执行 —— 从源头杜绝问题）

PPT 文案本就不该超长。截断的根治在数据层，不在渲染层：

- [ ] 每个 chip 一个**短词**：≤ 4~5 个汉字 / 一个英文术语。
- [ ] 「A / B」「A + B」这类合并项 → **拆成多个独立 chip**（同类就多放
      几个 chip，不要硬塞一个）。例：`ItemCF / UserCF` → 两个 chip。
- [ ] 英文用业内通用**简写**：DSSM、MMoE、ANN、DCN…（不写全称）。
- [ ] 英文长公司名 → 业内通用**中文简称**：`Momenta` → `魔门塔`、
      `Pony.ai` → `小马`（与同卡其他公司中文名风格一致）。
- [ ] landscape-map 结构铁律：**每个 tier 的 `--span` 合计必须 == 12**，
      否则 grid 折行 → 大片留白。改 span 后逐 tier 求和验证。

## 三、渲染自检清单（生成第一版后必做 —— 看完整全图，逐条核）

用 Playwright 渲染（注入 `:root{--fit:1}`、隐藏 nav），**等所有 img.complete
后截完整全页 1600×900 图**，然后逐条肉眼核对：

- [ ] **三层均匀分布、撑满**：tier 三等分占满区域（约 ~74% slide 高），
      不挤在上半部、底部无大片留白。（`.diagram-landscape`:
      `flex:1 1 auto; display:grid; grid-template-rows:1fr 1fr 1fr`）
- [ ] **三层间距正常**：层间有清晰间隔（`--tier-gap:22px` + `row-gap`），
      不贴在一起。
- [ ] **分割虚线在层间隙中央**：`.lm-row+.lm-row::before` 用
      `top:calc(-1*var(--tier-gap)/2)`，不是死值 margin-top。
- [ ] **全页 chip 同长同宽**：所有 `.lc-body` 的 chip 一个统一宽度（取全页
      最宽、但不超最窄列可用宽）；所有 `.ls-body`（嵌套小格）另一个统一宽。
      JS 只统一宽度，**字号始终是 CSS 固定值不变**。
- [ ] **零文字截断**：肉眼逐 tier 看，没有"小··""Mo··"这种被裁的。
- [ ] **logo 真加载**：截图前 img 全部 complete；个别公司 Clearbit 无收录
      回退首字母色块是正常的，不算 bug。
- [ ] **无 chip 重叠、无纵向溢出、无 JS 报错**（pageerror 为空）。
- [ ] **CSS 真生效自检**：computed `display` 应为 `grid` 而非 `block`。
      若是 block → 多半是规则 4 的 `<style>` 字面量导致 CSS 解析崩了。

## 四、LLM 二次复查流程（自检发现问题时）

自适应/渲染逻辑**不要改**（它是稳定的）。问题几乎都出在数据：

1. 全图发现某 chip 截断 / 某卡过挤 → **回到第二节数据规范**，精简该卡文案、
   拆 chip、英文换简写，**重新 build**。不要去改渲染 CSS/JS。
2. 改完数据后**重新执行第三节渲染自检全清单**（再看一次完整全图）。
3. 直到全图肉眼确认所有项通过，才算完成。

## 一句话总结

**数据层规范内容（短词/拆块/简写）+ 渲染层固定尺寸（不靠 JS 自适应）
+ 验证只信完整全图肉眼（不信局部图和自写检测）。** 这三条守住，
就不会重蹈那十几轮的覆辙。
